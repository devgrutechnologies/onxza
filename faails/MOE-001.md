# ONXZA MoE Precision Execution Architecture

> *Imagined by Aaron Gear. Created by Aaron Gear and Marcus Gear (AI Co-Creator). Powered by DevGru US Inc. DBA DevGru Technology Products. Using Powerful Anthropic Models, OpenAI Models, and Local LLMs.*

ONXZA implements a Mixture of Experts architecture at the agent orchestration layer.

## Principle

No generalist agent tries to do everything. Every task is routed to the specific expert agent built for that exact type of work. Each expert has its own:

- Configured model (best fit for domain)
- Skill MDs (domain-specific knowledge)
- Scripts (domain-specific automation)
- Memory (domain-specific learning)
- Verification criteria

This is model-agnostic, observable, auditable, and self-improving. Unlike internal MoE models, every expert decision is visible, traceable, and improvable by humans and agents alike.

---

## Execution Loop

### Step 1 — Task Arrives

Task enters via ticket system. PM assigns to appropriate expert agent.

### Step 2 — Routing Check

DTP_ONXZA_Router reviews task. Suggests optimal model based on task type. Logs suggestion.

### Step 3 — Expert Receives Task

Expert agent reads full ticket. Expert uses ITS OWN configured model.

- **Router suggestion matches expert model:** Execute immediately. No friction.
- **Router suggestion differs from expert model:** Log the difference. Expert executes with its own model anyway. Outcome recorded for both models to learn from. Data decides what is right over time. No overrides. No blocks. Just data.

### Step 4 — Sandboxed Execution

Expert executes task. If coding: DTP_ONXZA_ScriptEngine runs scripts in complete isolation. Output captured. Nothing touches production. Expert produces confidence score 0–100.

### Step 5 — FVP Verification

See FVP-001.md.

### Step 6 — Learning Capture

Pass or fail: Expert updates its skill MD. Router logs outcome against its suggestion. Verification agent logs the pattern. If 3+ agents fail the same check: flag to MG_Parent_AgentDeveloper to update the global standard.

## Model Selection by Expert Type

| Expert Type | Model |
|---|---|
| Autonomous coding loops | Claude Code |
| Complex strategy and vision | Claude Opus / Sonnet |
| Structured complex output | GPT-4o |
| Real-time and social research | Grok |
| Simple reasoning, well-defined | Local LLM first |
| Domain-specific tasks | ONXZA-LLM when trained |
| All other tasks | Cheapest capable model |

---

## Cost Principle

Always use the least expensive model that produces correct output. Local LLMs are tried first for ALL tiers if the task is within capability window. Cloud models used only when local cannot do the job to standard.

Tier 1 does NOT mean cloud model required. Tier 1 means reasoning is required. Local LLMs can handle Tier 1 tasks within their capability.

---

## Three Tiers — Corrected

**Tier 1 — Reasoning Required**
Any model capable of the reasoning needed. Router suggests. Expert decides. Local LLM is always a valid Tier 1 model for tasks within its capability.

**Tier 2 — Script + Model Hybrid**
Script handles mechanics. Any model handles the judgment layer. Local LLM preferred if capable. Escalate to cloud only if needed.

**Tier 3 — Pure Script / Cron**
Zero LLM involvement. Python / shell / n8n workflow. Fastest. Cheapest. Most reliable. Goal: push everything possible here over time.
