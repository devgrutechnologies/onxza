## Overview

Every output produced by any ONXZA agent MUST pass through this protocol before being accepted as complete. The FVP is the quality backbone of the FAAILS framework — it ensures that no unverified, inaccurate, or machine-sounding output ever reaches a human stakeholder or external system.

The protocol operates as a sequential 5-step loop with defined iteration limits, structured verification reports, and clear escalation paths.

---

## Step 1 — Confidence Self-Assessment

The producing agent MUST generate a confidence score between 0 and 100 alongside every output.

### Confidence Score Calculation Guidelines

The score is computed as a weighted average of four factors:

| Factor | Weight | Description |
|---|---|---|
| **Completeness** | 30% | Does the output address all acceptance criteria in the ticket? |
| **Factual Certainty** | 30% | Are all claims verifiable? Are sources available? |
| **Quality Fit** | 20% | Does the output match the expected format, tone, and depth? |
| **Self-Consistency** | 20% | Is the output internally consistent with no contradictions? |

Each factor is scored 0–100 by the producing agent. The weighted sum produces the final confidence score.

**Formula:** `confidence = (completeness × 0.30) + (factual_certainty × 0.30) + (quality_fit × 0.20) + (self_consistency × 0.20)`

### Decision Thresholds

- Score **< 70:** The agent MUST loop back and revise before submission. Loop count increments by 1.
- Score **>= 70 and < 85:** Proceed to Step 2. Output is flagged for close inspection.
- Score **>= 85:** Proceed to Step 2. Standard verification path.

Agents MUST NOT artificially inflate confidence scores. Persistent score inflation (detected when FVP fail rate exceeds 30% for outputs scored >= 85) SHOULD trigger a model recalibration flag in the routing log.

## Step 2 — Humanization Check

DTP_ONXZA_Verification MUST review the output for natural, human-quality presentation.

### Humanization Checklist

The verifier MUST check each applicable item:

- [ ] **No AI tells:** No phrases like "certainly," "I'd be happy to," "as an AI," "delve into," or similar markers
- [ ] **Natural sentence variation:** Mix of short and long sentences; no robotic uniformity
- [ ] **Active voice preferred:** Passive voice used only where appropriate (scientific writing, formal specs)
- [ ] **No filler paragraphs:** Every paragraph adds substance; no padding or restating the obvious
- [ ] **Consistent tone:** Matches the intended audience and document type
- [ ] **Proper formatting:** Headers, lists, and tables used appropriately — not excessively
- [ ] **Code quality (if applicable):** Clean, readable, properly commented, follows project conventions
- [ ] **No hallucinated citations:** Any referenced source MUST exist and be verifiable

**PASS →** Proceed to Step 3.
**FAIL →** Return to producing agent with specific failing items noted. Loop count increments by 1.

---

## Step 3 — Fact and Accuracy Verification

The verifier MUST confirm factual accuracy using methodology appropriate to the output type.

### Output-Type-Specific Methodology

**Written Content (articles, reports, documentation):**
- All factual claims MUST be verified against primary sources
- Statistics MUST include source and date
- Technical claims MUST be validated against official documentation
- Opinions MUST be clearly labeled as such

**Code Output:**
- Code MUST compile/execute without errors in the target environment
- All tests MUST pass
- Dependencies MUST be declared and version-pinned
- Security: no hardcoded secrets, no known vulnerable dependencies

**Data and Analytics:**
- Calculations MUST be independently reproducible
- Data sources MUST be cited
- Statistical methods MUST be appropriate for the dataset
- Outliers MUST be acknowledged and explained

**Configuration and Infrastructure:**
- Configs MUST be valid for the target system
- No secrets in plaintext
- Rollback procedures MUST be documented where applicable

### Additional Checks (All Types)

- Output MUST align with company vision.md
- Output MUST meet all acceptance criteria defined in the originating ticket
- Output MUST NOT contradict existing published ONXZA specifications

**PASS →** Proceed to Step 4.
**FAIL →** Return to producing agent with specific failures documented. Loop count increments by 1.

---

## Step 4 — Max Iteration Rule

If the cumulative loop count reaches **3** without passing all verification steps:

1. The verifier MUST create an escalation ticket to the assigned PM.
2. The escalation ticket MUST include:
   - Full loop history (all iterations with timestamps)
   - All output versions produced
   - All verification reports generated
   - Specific failure patterns identified
   - Recommended action (one of: retry with different model, escalate to human, accept with documented caveats)
3. The PM MUST decide the resolution path within the escalation ticket.

**No infinite loops. Ever.** Three iterations is the hard ceiling. If an agent cannot produce passing output in three attempts, the task requires human intervention or a different approach — not a fourth try at the same thing.

### Per-Step Time Limits

Each verification step MUST complete within defined time bounds:

| Step | Max Duration | Escalation on Timeout |
|---|---|---|
| Step 1 — Confidence Check | 30 seconds | Auto-fail; re-queue task |
| Step 2 — Humanization Check | 5 minutes | Flag to PM; proceed with warning |
| Step 3 — Fact Verification | 15 minutes (written), 10 minutes (code), 20 minutes (data) | Flag to PM; partial verification noted |
| Step 4 — Escalation Creation | 2 minutes | Auto-create minimal escalation |
| Step 5 — Acceptance | 1 minute | Auto-accept if Steps 1–3 passed |

If any step exceeds its time limit, the system SHOULD log the timeout and proceed with the defined escalation action. Time limits MAY be adjusted per-company via configuration.

---

## Step 5 — Acceptance

Output accepted. The system MUST:

1. Update the originating ticket with a completion note including the final confidence score
2. Attach the verification report to the ticket
3. Capture learnings: all involved agents (producer, verifier, router) MUST update their respective memory files
4. Log the complete FVP pass record for MPI aggregation (see MPI-001)

## Protocol Guarantees

This protocol IS the quality standard. Every output that leaves ONXZA has passed it.

- **Consistency:** The same verification criteria apply to every agent, every task, every time.
- **Transparency:** Every verification decision is logged and auditable.
- **Improvability:** Verification patterns feed back into the MPI (MPI-001) and routing optimization (ROUTING-001).
- **Trustworthiness:** External stakeholders can inspect the verification trail for any output.

This is what makes ONXZA trustworthy at scale. This is what makes FAAILS an industry standard.
