## Overview

The ONXZA Model Performance Index (MPI) is the first real-world benchmark of AI model performance in autonomous agentic workflows with end-to-end verification.

Every other benchmark tests models on static tasks evaluated by humans or automated rubrics against known answers. MPI tests models on real autonomous tasks — tasks that require planning, execution, iteration, and verification — evaluated by the FVP verification loop. This is production data, not lab data.

The MPI produces a single composite score per model, broken down by task type, alongside detailed performance profiles that reveal where each model excels and where it struggles.

---

## What It Measures

### Core Metrics

Every MPI evaluation captures six core metrics per model:

| Metric | Description | Unit |
|---|---|---|
| **FVP First-Pass Rate** | Percentage of tasks passing FVP on the first loop iteration | % (0–100) |
| **Average Loop Count** | Mean number of FVP iterations required to pass, by task type | Count (1.0–3.0) |
| **Time to Completion** | Median wall-clock time from task assignment to FVP acceptance | Seconds |
| **Cost per Completion** | Median token cost for a successful task completion | USD |
| **Task-Type Affinity** | Per-task-type pass rate revealing model strengths | % per type |
| **Failure Mode Distribution** | Categorized reasons for FVP failures (humanization, accuracy, completeness) | % per category |

### Derived Metrics

- **Efficiency Ratio:** FVP first-pass rate divided by cost per completion — measures quality per dollar
- **Reliability Index:** Percentage of tasks completed without escalation (loop count ≤ 3)
- **Versatility Score:** Standard deviation of pass rates across task types — lower is more versatile

## MPI Score Calculation Formula

The MPI composite score produces a single number (0–100) for each model, enabling direct comparison.

### Formula

```
MPI = (first_pass_rate × 0.35) + (efficiency_norm × 0.25) + (speed_norm × 0.20) + (cost_norm × 0.15) + (versatility_norm × 0.05)
```

### Component Definitions

**first_pass_rate** (weight: 35%)
Direct FVP first-attempt pass rate. Raw percentage, no normalization needed. This is the most important signal — can the model get it right the first time?

**efficiency_norm** (weight: 25%)
Normalized efficiency ratio. The model with the highest efficiency ratio in the benchmark set receives 100; all others are scaled proportionally.

`efficiency_norm = (model_efficiency / max_efficiency) × 100`

**speed_norm** (weight: 20%)
Normalized speed. The fastest model (lowest median completion time) receives 100.

`speed_norm = (fastest_median_time / model_median_time) × 100`

**cost_norm** (weight: 15%)
Normalized cost efficiency. The cheapest model (lowest median cost per completion) receives 100.

`cost_norm = (cheapest_median_cost / model_median_cost) × 100`

**versatility_norm** (weight: 5%)
Normalized versatility. The model with the lowest standard deviation in cross-task-type pass rates receives 100.

`versatility_norm = (lowest_std_dev / model_std_dev) × 100`

### Score Interpretation

| MPI Score | Rating | Interpretation |
|---|---|---|
| 90–100 | Exceptional | Consistently excellent across all dimensions |
| 75–89 | Strong | Reliable for most agentic workloads |
| 60–74 | Adequate | Suitable for specific task types; weaknesses in others |
| 40–59 | Below Average | Significant gaps; use only for narrow specializations |
| 0–39 | Poor | Not recommended for autonomous agentic workflows |

## Benchmark Methodology

### Reproducible Test Setup

To ensure MPI scores are comparable across time and across ONXZA installations, benchmark runs MUST follow a standardized methodology.

**Task Corpus:**
The benchmark uses a fixed corpus of 50 standardized tasks, distributed across all 10 defined task types (5 tasks per type). Each task has:
- A complete ticket with acceptance criteria
- A known-good reference output (for fact verification calibration)
- A defined complexity rating

**Task Types in Corpus:**
`code_generation`, `code_review`, `documentation`, `research`, `analysis`, `configuration`, `creative_writing`, `data_processing`, `strategy`, `communication`

**Environment Requirements:**
- Each model MUST be tested in isolation — no cross-model contamination
- Agent memory MUST be reset between benchmark runs (no learned advantages)
- FVP verification MUST use the same verifier model across all benchmark subjects
- Network conditions SHOULD be controlled (stable connection, no throttling)
- Temperature and sampling parameters MUST be documented per model

**Run Protocol:**
1. Reset agent memory and routing history
2. Load standardized task corpus
3. Execute all 50 tasks sequentially per model
4. Record full routing decision logs and FVP verification reports
5. Compute MPI score from aggregated outcomes
6. Repeat for each model under evaluation

### Model Comparison Fairness Methodology

To ensure fair comparison between models:

- **Prompt parity:** All models receive identical prompts. No model-specific prompt engineering.
- **Token budget parity:** All models are given the same maximum token budget per task. If a model requires significantly more tokens, this is reflected in the cost metric — not hidden.
- **Retry parity:** All models follow the same FVP loop rules (max 3 iterations). No model receives extra attempts.
- **Version pinning:** Model versions MUST be recorded. If a provider updates a model mid-benchmark, the run MUST be restarted or the version change documented.
- **No cherry-picking:** All 50 tasks MUST be included in the final score. Excluding tasks that a model failed is not permitted.

## Statistical Significance Requirements

MPI scores MUST meet minimum statistical standards before publication.

### Minimum Sample Sizes

- **Per-model minimum:** 50 tasks (the full benchmark corpus). Partial runs MUST NOT be published.
- **For trend claims ("Model X improved"):** Minimum 2 complete benchmark runs at different dates.
- **For ranking claims ("Model X outperforms Model Y"):** The difference in MPI scores MUST exceed the margin of error.

### Confidence and Error

- MPI scores MUST be reported with 95% confidence intervals.
- The confidence interval is computed using bootstrap resampling (1,000 iterations) over the per-task scores.
- If two models' 95% confidence intervals overlap, they MUST be reported as "statistically indistinguishable" rather than ranked.

### Anomaly Handling

- If a model fails more than 80% of tasks in a single task type, that task type MUST be flagged as an anomaly in the report.
- Anomalous results are included in the MPI score (no exclusion) but are highlighted in the detailed breakdown.
- If a model times out on more than 20% of tasks, the benchmark run SHOULD be flagged as potentially invalid and re-evaluated.

## Publication Format and Cadence

### Cadence

| Publication Type | Frequency | Audience |
|---|---|---|
| **Internal Dashboard** | Real-time | ONXZA agents, DevGru team |
| **Monthly Report** | First Monday of each month | Public (blog + docs site) |
| **Quarterly Deep Dive** | Every 3 months | Public (white paper format) |
| **Ad-hoc Alert** | As needed | Internal — when a model shows sudden degradation |

### Publication Format

Monthly reports MUST include:
1. Overall MPI rankings with confidence intervals
2. Task-type breakdown for top 5 models
3. Month-over-month trend for each model
4. Notable changes (new models entering, existing models improving/degrading)
5. Methodology notes (any corpus changes, environment changes)

Quarterly deep dives MUST additionally include:
- Statistical analysis of routing optimization improvements
- Cost savings from routing intelligence
- Model provider comparison (aggregate by provider)
- Recommendations for the ONXZA community

### Data Integrity

- Published MPI scores MUST be reproducible from the raw benchmark logs.
- Raw logs MUST be archived for the retention period defined in ROUTING-001.
- Any corrections to published scores MUST be issued as an erratum with explanation.
