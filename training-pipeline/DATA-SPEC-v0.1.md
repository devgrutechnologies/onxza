# ONXZA-LLM v0.1 — Training Data Collection Spec

**Teams:** dtp-onxza-llm (training), dtp-onxza-modelindex (collection)  
**Target model size:** 7B parameters (Mistral/Llama architecture)  
**Training target date:** M3 Sprint 10 (Day 150)  
**Collection start:** Immediate (parallel to M1/M2 CLI work)

---

## Model Target

ONXZA-LLM v0.1 should outperform general-purpose 7B models on three specific tasks:

1. **FAAILS compliance checking** — Given an `AGENTS.md` or ticket file, identify protocol violations
2. **Multi-agent routing decisions** — Given a task description, recommend optimal agent assignment and automation tier
3. **Ticket classification** — Given free-form task text, produce structured ticket frontmatter (`assigned_to`, `priority`, `type`, `vision_goal`)

General models (Llama-3, Mistral-7B) perform these tasks poorly because they have no training signal on FAAILS protocol specifics, ONXZA agent naming conventions, or the ticket schema. ONXZA-LLM v0.1 fixes this gap.

**What ONXZA-LLM does NOT need to do:**
- General coding (defer to larger models)
- Creative writing
- Math reasoning
- Broad world knowledge

The model is a specialist, not a generalist. This enables a smaller parameter count with higher task-specific accuracy.

## Data Categories

### Category 1: FAAILS Compliance Examples

**Purpose:** Train the model to read agent files and identify protocol violations.

**Source:**
- Primary: Real agent AGENTS.md files from active ONXZA installations (sanitized)
- Secondary: Synthetic examples generated by templating known-good files and introducing controlled violations

**Format (JSONL, one example per line):**
```json
{
  "type": "faails_compliance",
  "input": {
    "file_type": "AGENTS.md",
    "content": "[full file content]"
  },
  "output": {
    "compliant": true,
    "violations": [],
    "spec_refs": ["FAAILS-001", "FAAILS-008"]
  }
}
```

For violation examples:
```json
{
  "type": "faails_compliance",
  "input": {
    "file_type": "AGENTS.md",
    "content": "[content with naming violation]"
  },
  "output": {
    "compliant": false,
    "violations": [
      {
        "spec": "FAAILS-001",
        "field": "agent_name",
        "issue": "Name does not follow Company_Department_Role pattern",
        "found": "researcher-agent",
        "expected_pattern": "[Company]_[Dept]_[Role]"
      }
    ]
  }
}
```

**Target volume:** 2,000 examples (60% compliant, 40% violation)  
**Quality gate:** Each violation example must have a real FAAILS spec citation. Synthetic examples reviewed by dtp-onxza-architect before inclusion.  
**Collection timeline:** Weeks 1–4 (Days 1–28)

### Category 2: Routing Decision Examples

**Purpose:** Train the model to assign tasks to the right agent and automation tier.

**Source:**
- Synthetic (primary for v0.1 — real routing data accumulates in M2)
- Generated from: task descriptions + agent registry + FAAILS-007 tier definitions

**Format:**
```json
{
  "type": "routing_decision",
  "input": {
    "task_description": "Research top 5 competitors to LangChain in the multi-agent space",
    "available_agents": [
      {"id": "dtp-onxza-architect", "role": "Architect", "domain": "system design"},
      {"id": "dtp-onxza-pm", "role": "PM", "domain": "product management"},
      {"id": "dtp-research-analyst", "role": "Research Analyst", "domain": "research"}
    ],
    "company_context": "DevGru Technology Products"
  },
  "output": {
    "recommended_agent": "dtp-research-analyst",
    "automation_tier": 1,
    "reasoning": "Task is research (Tier 1, no irreversible action). Research Analyst role matches domain. PM and Architect are out of scope.",
    "confidence": 90
  }
}
```

**Target volume:** 3,000 examples  
**Quality gate:** Each routing decision must have a `reasoning` field that cites a FAAILS spec or role definition. Routing must be internally consistent across similar examples.  
**Collection timeline:** Weeks 1–8 (Days 1–56) — ongoing synthetic generation

### Category 3: Ticket Classification Examples

**Purpose:** Train the model to convert free-form task descriptions into structured ONXZA ticket frontmatter.

**Source:**
- Sanitized real DTP tickets (remove company-sensitive context, preserve structure)
- Synthetic examples based on ticket type templates

**Format:**
```json
{
  "type": "ticket_classification",
  "input": {
    "raw_text": "Someone needs to fix the bug where onxza agent list crashes when no agents are registered",
    "company": "DTP",
    "available_agents": ["dtp-onxza-cli", "dtp-onxza-architect", "dtp-onxza-pm"]
  },
  "output": {
    "title": "Fix: `onxza agent list` crashes when agent registry is empty",
    "type": "bug_report",
    "assigned_to": "dtp-onxza-cli",
    "priority": 2,
    "vision_goal": "Product shipping — CLI reliability is critical for M1 release.",
    "tags": ["cli", "bug", "agent-list", "m1"]
  }
}
```

**Target volume:** 2,500 examples (1,500 real sanitized, 1,000 synthetic)  
**Quality gate:** `assigned_to` must be a valid agent ID. `vision_goal` must reference an active roadmap milestone. `priority` must be 1–3.  
**Collection timeline:** Weeks 1–6 (Days 1–42)

### Category 4: Multi-Agent Conversation Traces

**Purpose:** Train the model to understand how agents communicate, escalate, and coordinate via the ticket system.

**Source:**
- Sanitized session transcripts from DTP operations (ticket creates, dispatches, result reports)
- Synthetic multi-turn conversations following FAAILS-002 protocol

**Format:**
```json
{
  "type": "agent_conversation",
  "turns": [
    {
      "agent": "dtp-coo",
      "action": "create_ticket",
      "content": "TICKET-20260322-DTP-DAILY-5: Write ONXZA Quickstart Guide",
      "ticket_metadata": {
        "assigned_to": "dtp-onxza-pm",
        "priority": 1,
        "type": "sprint_ticket"
      }
    },
    {
      "agent": "dtp-onxza-pm",
      "action": "acknowledge_ticket",
      "content": "HEARTBEAT received. Reading MEMORY.md. Executing DAILY-5 via THREE-PHASE TASK LOOP. PHASE 1: writing plan."
    },
    {
      "agent": "dtp-onxza-pm",
      "action": "complete_ticket",
      "content": "DAILY-5 complete. quickstart.mdx written, TORI-QMD PASS. Results at workspace/results/DAILY-5-results.md."
    }
  ],
  "outcome": "success",
  "faails_specs_demonstrated": ["FAAILS-002", "FAAILS-008"]
}
```

**Target volume:** 1,500 multi-turn traces  
**Quality gate:** Each trace must demonstrate at least one FAAILS spec in action. Traces with ambiguous outcomes are excluded.  
**Collection timeline:** Weeks 4–12 (ongoing from M2 operations)

## Collection Pipeline

**Tooling:** Existing `training-pipeline/` scripts (run-pipeline.js, collect.js, anonymize.js, filter.js, format.js)

**Storage:**
- Raw collected data: `training-pipeline/output/raw/`
- Anonymized: `training-pipeline/output/anonymized/`
- Final JSONL: `training-pipeline/output/training-data.jsonl`

**Quality checks before each batch:**
1. Schema validation (all required fields present)
2. Anonymization verification (no PII, company names, API keys)
3. FAAILS spec citation check (all spec references are valid)
4. Deduplication (cosine similarity > 0.95 = duplicate, exclude)

**Manifest:** `training-pipeline/output/manifest.json` — tracks total examples per category, last collection date, quality gate pass rate.

## Total Data Target for v0.1 Training Run

9,000 high-quality, domain-specific examples is sufficient for a task-specific fine-tune on a 7B base model. Target: ≥85% accuracy on FAAILS compliance checking, ≥80% on routing decisions (validated against human-labeled hold-out set of 500 examples).
